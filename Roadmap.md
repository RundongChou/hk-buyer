# 香港买手跨境代购 Roadmap（10 Sprints）

## 0. 北极星基准（原始产品说明文档）

以下内容完整作为本项目的北极星约束与需求上限，后续所有 Sprint 需求若与本章冲突，以本章为准。

### 0.1 产品定位
1. 面向内地用户，提供香港正品商品（药房、药妆、零售）的一站式下单与跨境履约服务。
2. 平台核心价值：正品保障、价格优势、可追踪履约、买手激励撮合。
3. 标准履约承诺：用户下单后 7-15 天签收（按合规清关路径）。

### 0.2 北极星目标与指标
1. NSM：30天跨境履约闭环订单数（CCO-30）。
定义：支付成功 + 买手接单 + 采购完成 + 清关出库 + 用户签收。

2. 护栏指标：

| 指标 | 目标 |
| --- | --- |
| 假货投诉率 | <= 0.3% |
| 超时未接单率（72h） | <= 15% |
| 订单取消率 | <= 8% |
| 7-15天履约达成率 | >= 90% |
| 合规清关成功率 | >= 99.5% |
| 用户NPS | >= 45 |

### 0.3 用户与角色
1. C端用户：浏览、下单、支付、追踪、售后。
2. 香港买手：接单、采购、上传凭证、交仓、结算。
3. 平台运营：选品、定价、活动、异常处理。
4. 平台风控：资质审核、交易风控、反作弊、合规审计。
5. 仓配与物流：入仓、分拣、清关、末端派送。
6. 财务：账务、分账、结算、发票与对账。

### 0.4 业务闭环
1. 用户选品下单支付。
2. 系统生成采购任务并在买手池发布。
3. 买手接单并采购，提交小票/凭证。
4. 商品交付香港集货仓，质检与入库。
5. 合规清关后发往内地。
6. 用户签收，平台分账结算。
7. 异常订单走售后与仲裁。

### 0.5 核心功能清单

#### 0.5.1 前台（用户端）
1. FR-U01 账号体系：手机号注册登录、实名认证、收货地址管理。
2. FR-U02 商品浏览：类目、品牌、关键词、筛选（价格/功效/库存）。
3. FR-U03 SKU详情：图文详情、规格、港币/人民币价格、预计时效、库存状态、正品说明。
4. FR-U04 购物车与下单：凑单、优惠券、运费估算、税费提示。
5. FR-U05 支付：微信/支付宝，支付状态回调与失败补偿。
6. FR-U06 订单中心：待接单/采购中/清关中/运输中/已签收全链路追踪。
7. FR-U07 售后中心：退货退款、少件破损、真伪争议。
8. FR-U08 消息通知：接单、发货、异常、退款、活动通知。
9. FR-U09 评价体系：商品评价、买手服务评分、平台服务评分。
10. FR-U10 推荐与搜索优化：热销、复购推荐、相似替代商品。

#### 0.5.2 前台（买手端）
1. FR-B01 入驻与审核：身份、资质、支付账户、信用评估。
2. FR-B02 任务大厅：按品类/利润/区域筛选任务，抢单与放弃。
3. FR-B03 采购执行：采购清单、门店建议、提货码、交仓码。
4. FR-B04 凭证上传：小票、发票、商品批次、有效期、实拍图。
5. FR-B05 履约管理：超时预警、缺货上报、替代建议。
6. FR-B06 收益中心：订单收益、服务费、结算周期、提现。
7. FR-B07 信誉体系：接单率、超时率、投诉率、信用分。

#### 0.5.3 管理后台
1. FR-A01 商品中心：SPU/SKU、图文、属性、禁售规则、上/下架。
2. FR-A02 价格中心：基础价、服务费率、动态加价策略、价格保护。
3. FR-A03 库存中心：可售库存、在途库存、缺货预警、锁库存。
4. FR-A04 订单调度：自动派单、人工干预、任务重派、合单拆单。
5. FR-A05 买手管理：准入审核、分层、黑白名单、处罚与激励。
6. FR-A06 仓配清关：入仓、质检、打包、关务资料、运单轨迹。
7. FR-A07 风控合规：反欺诈、真伪抽检、敏感品类拦截、审计日志。
8. FR-A08 客服工单：退款、纠纷、补偿、仲裁结论。
9. FR-A09 财务结算：分账、对账、手续费、技术服务费、税务报表。
10. FR-A10 数据看板：转化漏斗、履约时效、毛利、复购、异常预警。

### 0.6 核心规则（关键）
1. 无人接单提价规则：任务发布后 72 小时无人接单，自动提价 `max(5%, 20元)`，每 24 小时最多一次，单订单最多提价 3 次，封顶 20%。
2. 缺货处理规则：买手上报缺货后可触发替代推荐；用户确认替代或部分退款。
3. 正品保障规则：凭证必传 + 仓端抽检 + 高风险品类加严审查。
4. 分账规则：用户支付金额按“商品成本、买手收益、物流成本、平台手续费/技术服务费”自动拆分。
5. SLA规则：接单 SLA 72 小时；全链路履约目标 7-15 天。

## 1. 北极星指标与商业化指标（重构版）

### 1.1 North Star Metric（NSM）
**30天跨境履约闭环订单数（CCO-30）**

闭环定义（同一用户 30 天内，至少 1 单满足）：
1. 支付成功；
2. 买手接单并上传采购凭证；
3. 仓配入库与清关出库；
4. 用户签收完成。

### 1.2 护栏指标（防偏航）
1. 假货投诉率 <= 0.3%。
2. 超时未接单率（72h） <= 15%。
3. 订单取消率 <= 8%。
4. 7-15天履约达成率 >= 90%。
5. 合规清关成功率 >= 99.5%。
6. 用户 NPS >= 45。

### 1.3 商业化核心指标
1. 订单毛利率（含技术服务费）>= 目标阈值。
2. 买手侧有效接单率与活跃买手数持续增长。
3. 30 天复购率持续提升。
4. 风险成本（赔付/拒付/假货争议）占比持续下降。

### 1.4 价值主张（从可商业化角度）
1. 用户侧：香港正品 + 价格优势 + 全链路可追踪，降低跨境购买决策成本。
2. 买手侧：透明利润与阶梯激励，提升接单意愿与履约稳定性。
3. 平台侧：撮合手续费 + 技术服务费 + 运营工具沉淀，形成可规模化模型。

## 2. 竞品研究结论（2026-03-14）

> 说明：以下基于公开检索可得信息（中文搜索、App Store、微信生态公开资料）整理，用于 Sprint 1 的功能取舍，不作为法律或财务结论。

### 2.1 竞品样本（网站/App/小程序）
1. **Buy&Ship（国际代购/转运）**
   - 公开强调“海外代购 + 集运 + 关税与物流规则透明”。
   - 启发：需要把“费用构成与运输规则”前置到用户下单决策。
2. **HKTVmall（香港电商平台）**
   - 公开强调“海量SKU、优惠活动、履约体验和会员权益”。
   - 启发：需要建立可持续的品类深度和运营活动系统，而非一次性代购。
3. **屈臣氏官方微信小程序（中国内地）**
   - 公开信息显示其小程序承载“新品首发、会员、优惠券、便捷下单”。
   - 启发：小程序端要优先保留高频交易能力和会员触达能力。
4. **持牌药房代购 App（如药房聚合类）**
   - 公开强调“药房品类覆盖、商品搜索、订单追踪”。
   - 启发：药妆/药品类目必须强化资质信息、适用说明、售后规则提示。
5. **京东国际（跨境电商）**
   - 公开规则强调“跨境商品正品保障、履约时效与售后规则明确”。
   - 启发：平台需要可审计的正品链路与明确售后边界。

### 2.2 竞品共性能力
1. 商品供给密度高（SKU 多、品牌全、活动频次高）。
2. 支付与履约状态展示清晰。
3. 会员与促销体系完整（券、满减、积分、专享价）。

### 2.3 当前缺口与可突破点
1. **可追溯不完整**：多数“代购”场景缺少凭证结构化与证据链查询。
2. **供给激励不系统**：买手接单靠人工经验，缺少自动加价与信誉协同。
3. **异常闭环不彻底**：缺货替代、部分退款、争议仲裁链路割裂。
4. **平台化弱**：缺少从买手、仓配、关务到财务的一体化后台。

### 2.4 Sprint 1 功能吸收与取舍策略
1. 吸收：搜索筛选、SKU详情、优惠触达、订单可视化、买手任务大厅。
2. 强化：采购凭证结构化、正品保障证据链、无人接单动态激励。
3. 延后：复杂会员分层、营销自动化、多渠道投放归因（放到 Sprint 7/9）。

## 3. 核心卖点（商业化导向）

1. **正品可证据化**：用户不仅看“正品承诺”，还能看采购凭证、抽检记录、批次信息。
2. **价格可解释**：订单展示“商品价 + 履约服务费 + 物流税费估算”，让价格优势透明可比。
3. **供给可激励**：无人接单自动提价 + 买手信誉分联动，提升接单率并控制履约时效。
4. **履约可追踪**：从接单、采购、入仓、清关到签收全链路状态可视化。
5. **平台可盈利**：撮合手续费 + 技术服务费 + 供应链协同效率提升，形成长期经营模型。

## 5. Sprint 1 PRD（基石 Sprint，详细版）

### 5.1 Sprint 名称
`Sprint 1：香港买手跨境代购最小交易闭环 MVP（竞品关键能力落地）`

### 5.2 Sprint 目标（必须达成）
1. 在 1 个 Sprint 内打通最小交易闭环：用户下单支付 -> 买手接单采购 -> 上传凭证 -> 订单状态可追踪。
2. 建立“正品可追溯”最小能力：每单至少 1 条采购凭证，可在后台审计。
3. 跑通任务发布与接单基础规则，支持 72h 未接单预警（自动提价逻辑可先 V0 上线）。
4. 形成首版事件埋点，能够计算 CCO-30 漏斗的前两段（支付、接单）。

### 5.3 竞品映射后的产品策略

#### 5.3.1 必须吸收的能力（Sprint 1 立刻做）
1. 商品可搜索、可筛选、可查看详情（对标 HK 电商体验）。
2. 下单支付体验完整且状态明确（对标跨境电商的交易稳定性）。
3. 买手任务大厅与抢单机制（对标代购平台供给组织方式）。
4. 消费者可见的履约状态节点（对标主流电商履约可视化）。

#### 5.3.2 必须差异化的能力（Sprint 1 就做）
1. 采购凭证结构化（小票、批次、有效期、实拍）而非仅图片上传。
2. 订单级“正品保障页”原型：展示凭证摘要、抽检状态、免责声明。
3. 无人接单预警 + 提价建议看板（运营可手动确认执行）。

#### 5.3.3 延后能力（避免 Sprint 1 过载）
1. 会员体系、积分、复杂促销规则。
2. 自动清结算与税务报表。
3. 多渠道（App/小程序）统一运营后台。

### 5.4 用户与核心场景

用户角色：
1. 内地用户（下单方）
2. 香港买手（履约方）
3. 运营人员（调度方）

核心场景：
1. 用户从 SKU 列表选择商品，完成下单支付并查看“待买手接单”。
2. 买手在任务大厅接单，采购后提交凭证并等待运营审核。
3. 运营审核凭证通过后，订单进入“待入仓/运输中”状态，用户可追踪。

### 5.5 范围（In Scope / Out of Scope）

In Scope：
1. 用户端：注册登录、商品浏览、购物车、下单支付、订单追踪。
2. 买手端：入驻申请、任务大厅、接单、凭证上传、收益预估。
3. 后台：商品管理、订单调度、凭证审核、异常标记。
4. 风控：基础黑名单、敏感品类拦截、审计日志。

Out of Scope：

### 1. 背景与问题定义
Roadmap Sprint 1 目标是“竞品对标最小交易闭环 MVP”。当前仓库仅有文档，尚未形成可运行的前后端分离工程，也没有可验证的下单-接单-凭证-审核-状态回传链路。

护栏映射：
1. 假货投诉率：通过结构化凭证字段与审核动作留痕降低假货争议风险。
2. 72h 超时未接单率：任务发布时写入接单截止时间，为超时统计和后续提价引擎提供数据基础。
3. 订单取消率：通过最小可追踪时间线提升用户可见性，降低因信息不透明导致的取消。
4. 7-15 天履约达成率：本 Sprint 不直接达成签收，但建立状态机与时间线，为后续时效优化提供可观测基础。
5. 合规清关成功率：仅保留合规清关状态扩展点，不实现非合规路径。
6. 用户 NPS：通过可追踪与凭证可审计增强信任，预期提升早期满意度。

## Roadmap Sprint 2 PRD（2026-03-14）

### 1. 背景与问题定义
Roadmap Sprint 2 的目标是“商品与库存中台 V1”。Sprint 1 已完成下单到凭证审核的最小交易链路，但商品侧仍是手工输入 SKU 与价格，缺少标准建品、上架审核、库存快照与缺货降级能力。当前问题会直接导致：
1. 无法规模化上新与维护价格策略；
2. 用户端无法稳定搜索可售商品；
3. 库存不足时存在误售风险，影响履约时效与取消率。

### 2. Sprint 目标（可验证）
1. 建立 SPU/SKU 与价格策略最小中台能力，支持后台建品与批量导入 SKU。
2. 建立 SKU 上架审核与发布状态管理，用户端仅可检索“审核通过且上架”的商品。
3. 建立库存快照与缺货降级能力：缺货 SKU 可见但不可下单，后台可查看缺货预警。
4. 下单链路接入商品价格与库存校验，确保“可搜可买且不误售”。

### 3. 非目标（本 Sprint 不做）
1. 不做复杂促销、优惠券、税费估算与支付补偿（Roadmap Sprint 3）。
2. 不做买手分层派单与信誉体系（Roadmap Sprint 4）。
3. 不做清关直连与物流轨迹对接（Roadmap Sprint 6）。
4. 不做任何绕关/走私能力，仅按合规清关路径保留后续扩展字段。

### 4. 用户故事与使用场景
1. 用户端（H5）：作为用户，我可以搜索 SKU，查看价格/库存状态，仅对可售 SKU 发起下单。
2. 管理后台：作为运营，我可以创建 SPU、批量导入 SKU、审核上架并调整库存。
3. 数据平台：作为运营分析，我可以看到 SKU 总量、已上架数量、缺货数量，判断供给健康度。

### 5. 范围（In Scope / Out of Scope）
In Scope：
1. 商品模型：SPU、SKU、价格策略、库存快照。
2. 管理操作：建品、批量导入、上架审核、库存调整、缺货预警查询。
3. 用户检索：SKU 搜索、可售状态展示、缺货降级展示。
4. 交易衔接：订单创建阶段进行 SKU 上架状态与库存校验，并按系统价格入单。

Out of Scope：
1. 多仓库存分配、采购在途库存（后续 Sprint）。
2. 高级搜索（同义词、拼写纠错、召回排序）。
3. 自动补货与智能定价。

### 6. 功能需求清单（FR）
1. FR-S2-01：后台可创建 SPU（名称、品牌、类目、审核状态）。
2. FR-S2-02：后台可按 SPU 批量导入 SKU（规格、展示价、服务费率、初始库存）。
3. FR-S2-03：后台可对 SKU 执行上架审核（APPROVE/REJECT），记录审核日志。
4. FR-S2-04：后台可调整 SKU 库存快照，并写入调整原因。
5. FR-S2-05：用户端可按关键词搜索 SKU，返回价格、库存、可售状态。
6. FR-S2-06：缺货 SKU 在用户端显示“缺货不可下单”，下单接口必须拒绝误售请求。
7. FR-S2-07：数据平台可查看商品供给指标（sku_total / sku_published / sku_out_of_stock）。

### 7. 数据与接口变更（MySQL）
1. 新增表：
   - `sku_spu`（SPU 主数据）
   - `sku_item`（SKU 主数据与发布状态）
   - `sku_price_policy`（价格策略：基础价、服务费率、当前到手价）
   - `stock_snapshot`（可售库存、锁定库存、预警阈值）
   - `sku_publish_audit_log`（上架审核留痕）
2. 新增索引：
   - `idx_sku_item_publish`（发布状态 + 更新时间）
   - `idx_sku_item_keyword`（名称 + 品牌）
   - `idx_stock_snapshot_qty`（可售库存 + 更新时间）
3. 迁移脚本：新增 `db/mysql/V2__sprint2_catalog_inventory.sql`。
4. API 变更（Spring MVC）：
   - `POST /api/v1/admin/catalog/spus`
   - `POST /api/v1/admin/catalog/skus/batch-import`
   - `POST /api/v1/admin/catalog/skus/{skuId}/publish-audit`
   - `POST /api/v1/admin/catalog/skus/{skuId}/stock-adjust`
   - `GET /api/v1/admin/inventory/alerts/out-of-stock`
   - `GET /api/v1/catalog/skus`
   - `GET /api/v1/catalog/skus/{skuId}`
   - `GET /api/v1/admin/metrics/catalog`
5. 与 Sprint 1 兼容策略：保留原下单接口结构；`unitPrice` 改为可选，优先使用商品中心价格策略计算，避免前端手工价格漂移。

### 8. 前端交互与页面变更（TypeScript + JSX）
1. H5（TSX）：新增 SKU 搜索区与结果列表，显示上架状态/库存状态；点击“用于下单”回填 SKU 与系统价格。
2. 管理后台（TSX）：新增建品、批量导入、上架审核、库存调整、缺货预警操作区。
3. 数据平台（TSX）：新增商品与库存健康指标卡。

### 9. 后端实现变更（Java 8 + Spring MVC）
1. 新增 `CatalogController` 与 `CatalogAdminController`，承载用户检索与后台商品管理接口。
2. 新增 `CatalogService`、`CatalogRepository`，封装建品、审核、库存、检索逻辑。

### 11. 风险与回滚方案
1. 风险：库存校验改造可能影响已有下单链路。
   回滚：保留原字段兼容，异常时可降级回旧价格计算并关闭库存强校验开关（代码开关位）。
2. 风险：后端测试环境缺失 Java/Maven，无法本地验证编译。
   回滚/缓解：记录阻塞并建议在 CI/JDK8 容器补跑。
3. 风险：SKU 检索性能在数据量增长后下降。
   回滚/缓解：先落关键索引，后续 Sprint 增加分页与检索优化。
